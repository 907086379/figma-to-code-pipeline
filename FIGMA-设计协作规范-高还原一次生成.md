# Figma 设计协作规范（精简关键版）

> 目标：让设计稿在本项目里尽量首轮高还原生成代码。

## 1) 五条关键红线（必须遵守）

- **结构可还原**：使用清晰分组 + Auto Layout，避免大量绝对定位与无语义 `Frame` 堆叠。
- **视觉值可映射**：颜色/字体/间距优先 Token 化；禁止“接近值”混用。
- **状态要齐全**：不能只交默认态；关键组件至少给默认、选中/未选中、展开/收起、禁用/错误（按业务有则给）。
- **交互要闭环**：跨页面/弹层必须写清 from -> to，分支与空态要显式给出。
- **命名可追溯**：页面/组件/分组使用业务语义命名，避免 `Frame 1`、`Group 2`。

## 2) 交付研发/Agent 的最小清单

- Figma 链接（必要时附 node-id）+ 本次改动范围。
- 状态清单（每个关键组件有哪些状态）。
- Token 变更清单（颜色、字体、间距）。
- 交互清单（跳转关系、分支条件、异常/空态）。
- 资源策略（图标来源、图片替换方案、是否允许远程资源）。

## 3) 提交前 60 秒自检

- 是否还存在“只有默认态，没有错误/禁用/展开态”的组件？
- 是否还有未 token 化的关键视觉值？
- 是否还有无语义命名（`Frame xxx`）影响映射？
- 是否所有关键流程都能从稿内直接看出，不靠口头解释？
- 是否让不熟悉业务的人也能按稿还原主要结构与行为？

## 4) 常见返工原因（禁止）

- 只交静态视觉，不交状态与交互。
- 同类组件在不同页面使用不同结构和间距体系。
- 文案/格式未定稿，用占位文本代替。
- 依赖临时资源但未给可落地替换方案。
